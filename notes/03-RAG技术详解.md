 # RAG 技术详解
 
 ## 为什么需要 RAG
 - **知识静态**：训练后知识冻结
 - **幻觉**：可能生成看似正确但错误的内容
 - **不可追溯**：不知道回答基于什么信息
 
 ## 完整流程
 用户问题 -> Query 改写 -> Embedding -> 向量检索 -> 获取上下文 -> Prompt 组装 -> LLM 生成 -> 回答
 
 ## 关键模块
 
 ### 文档分块
 | 策略 | 优点 | 缺点 | 适用场景 |
 |------|------|------|----------|
 | 固定长度 | 简单快速 | 可能切断语义 | 不敏感内容 |
 | 递归字符 | 保留语义 | 块大小不一 | 一般文档 |
 | 语义分割 | 效果最好 | 计算成本高 | 高质量场景 |
 经验值：chunk_size 500-1000 字符，overlap 100-200 字符。
 
 ### Embedding
 - text-embedding-3-small：1536 维，性价比高
 - text-embedding-3-large：3072 维，效果更好
 - bge-m3：国产，对中文友好
 
 ### 向量检索
 - **余弦相似度**：最常用
 - **HNSW**：最主流的 ANN 算法
 - **Hybrid Search**：向量检索 + 关键词检索
 
 ## 常见优化
 1. Query 改写：把问题改写为更适合检索的形式
 2. Hybrid Search：向量 + 关键词
 3. Reranking：初筛后用更精确的模型排序
 4. Self-RAG：让模型自己判断是否需要检索
 
 ## 从后端视角理解 RAG
 RAG 本质上是一个数据流管道：Input -> Transform -> Store -> Search -> Transform -> Output。
 区别在于数据变成了向量，搜索变成了语义相似度，拼装逻辑变成了 Prompt + LLM。
 你在数据管道、缓存、API 设计方面的经验完全适用。
