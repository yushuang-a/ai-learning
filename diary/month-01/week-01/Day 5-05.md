# Day 5: 错误处理与可靠性

**日期**：2026-04-25

## 今天做了什么

LLM API 的错误处理比传统 API 更复杂。Rate Limit、Token 超限、Timeout 是最常见的三种错误。

实现了一个带指数退避的 retry 机制：
- 429（Rate Limit）：等待 1s、2s、4s...最多 5 次
- 500（Server Error）：重试 3 次
- 其他：直接抛出

**经验**：写了一个通用的 `request_with_retry()` 装饰器，可以在后续所有 API 调用中复用。从后端工程的角度看，这个抽象做得值。

还研究了 `max_tokens` 和 `temperature` 对输出的影响。Temperature 从 0.1 测到 0.9，差异非常明显——难怪文档推荐事实性任务用低值。
